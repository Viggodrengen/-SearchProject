SearchProject projektordbog

Denne ordbog er lavet som et hurtigt opslagsværk til eksamen. Den tager kun de begreber med, som er vigtige i jeres fremlæggelse, og forklarer både hvad de er, hvordan I bruger dem i projektet, og hvordan de kan siges på en enkel måde til lærerne.

Pointen er ikke at kunne snakke mest teknisk. Pointen er at kunne forklare roligt, hvorfor hvert element findes i arkitekturen, og hvilken værdi det giver.

Sådan bruger I den

Hurtigt overblik

Hvis I kun har fem minutter, så læs først de gule bokse med “Sig det sådan”. De er skrevet som korte eksamenssvar. Hvis I har lidt mere tid, så brug kortene som repetition af de vigtigste arkitekturbegreber i projektet.

1. Forstå systemet
Bruger sender søgning gennem webapp eller CLI til Nginx og videre til SearchApi.
2. Se hvor fart kommer fra
Redis bruges til at cache gentagne søgeresultater, så API'et ikke skal beregne det samme igen.
3. Se hvor data kommer fra
Indexer bygger søgedata i Postgres, og SearchApi læser dem ved forespørgsler.
4. Se hvor drift kommer fra
Prometheus, Grafana og Loki gør det muligt at følge hastighed, fejl og cache-adfærd.
Arkitektur og skalering

Begreber om systemets opbygning

A1Nginx

Hvad er det?

Nginx er en reverse proxy, som står foran jeres API-instanser og modtager trafikken først.

Hvordan bruger vi det?

I Docker Compose sender webappen og CLI'en requests til Nginx, og Nginx fordeler dem videre til searchapi1 og searchapi2.

Sig det sådan:
Nginx er indgangen til vores backend. Det gør det nemt at fordele trafik mellem flere API-instanser uden at bygge load balancing ind i selve applikationskoden.

A2Stateless service

Hvad er det?

En stateless service gemmer ikke vigtig bruger- eller sessionsdata lokalt i sin egen proces mellem requests.

Hvordan bruger vi det?

SearchApi holder ikke søgetilstand selv. Varig data ligger i Postgres, og midlertidige søgeresultater ligger i Redis.

Sig det sådan:
Vores API er stateless, og det er vigtigt, fordi det gør det lettere at starte flere ens instanser og fordele trafik mellem dem.

A3X-skalering

Hvad er det?

X-skalering betyder, at man kører flere kopier af den samme service i stedet for kun at gøre én maskine større.

Hvordan bruger vi det?

I projektet kører der to SearchApi-instanser bag Nginx. Det er den tydeligste skaleringsmekanisme, I har implementeret.

Sig det sådan:
Vi skalerer udad ved at have flere API-replikaer, så vi både kan håndtere mere trafik og tåle, at én instans går ned.

A4Y-skalering

Hvad er det?

Y-skalering betyder funktionel opdeling, altså at systemet deles op i forskellige services med hver deres ansvar.

Hvordan bruger vi det?

Jeres løsning er delt op i SearchWebApp, SearchApi, Indexer, Redis, Postgres, Nginx og observability-komponenter.

Sig det sådan:
Vi har delt ansvaret op mellem flere komponenter, så UI, søgelogik, dataopbygning, cache og driftsoverblik ikke ligger blandet sammen ét sted.

A5Failover

Hvad er det?

Failover betyder, at systemet kan fortsætte med at svare, selv om en del af det fejler.

Hvordan bruger vi det?

Når én SearchApi-instans stopper, kan Nginx stadig sende trafik til den anden. Det er derfor, to replikaer giver mere end bare performance.

Sig det sådan:
Failover i vores projekt betyder, at brugeren stadig kan få svar, selv hvis én API-instans går ned under drift.

A6Docker Compose

Hvad er det?

Docker Compose er værktøjet, der starter hele jeres lokale driftsmiljø med flere containere på én gang.

Hvordan bruger vi det?

Compose starter Postgres, Redis, Indexer, SearchApi x2, Nginx, SearchWebApp, Prometheus, Grafana og Loki.

Sig det sådan:
Compose er vores implementerede driftsetup nu. Det giver os et samlet miljø, der minder om rigtig drift, uden at vi behøver Kubernetes for at demonstrere arkitekturen.

A7Kubernetes

Hvad er det?

Kubernetes er en platform til at køre og styre containere i større, mere produktionsnære miljøer.

Hvordan bruger vi det?

I jeres projekt er Kubernetes ikke det nuværende runtime-setup, men den er vist som den naturlige næste driftsretning i diagrammerne.

Sig det sådan:
Vi påstår ikke, at Kubernetes er implementeret nu. Vi bruger det som en produktionsretning, fordi arkitekturen allerede er delt op i services, som passer godt til den model.
Data, søgning og cache

Begreber om hvor data ligger, og hvorfor søgning går hurtigere

D1Postgres

Hvad er det?

Postgres er den persistente database i projektet. Den holder den varige data, som systemet bygger søgning på.

Hvordan bruger vi det?

Indexer skriver søgedata til Postgres, og SearchApi læser derfra, når et resultat ikke kan hentes fra cache.

Sig det sådan:
Postgres er vores kilde til sandhed. Redis er kun et hurtigt mellemlag oven på den.

D2Indexer

Hvad er det?

Indexer er den proces, der bygger eller genopbygger søgedata, så API'et ikke selv skal gøre det under hver forespørgsel.

Hvordan bruger vi det?

I Compose kører Indexer som en separat container før SearchApi starter. Efter indeksering rydder den også Redis-cachen, så gamle søgeresultater ikke bliver hængende.

Sig det sådan:
Indexer forbereder dataene til søgning. Den adskiller dataopbygning fra selve request-flowet og gør arkitekturen mere ren.

D3Omvendt indeks

Hvad er det?

Et omvendt indeks er en datastruktur, hvor man går fra ord til dokumenter i stedet for fra dokument til ord.

Hvordan bruger vi det?

Hele idéen med jeres søgesystem er, at ord kan slås op effektivt, så relevante dokumenter findes hurtigere end ved lineær gennemlæsning hver gang.

Sig det sådan:
Det omvendte indeks er det, der gør søgningen mulig på en effektiv måde. I stedet for at læse alle dokumenter igennem hver gang, slår vi ord op i en struktur, der allerede er forberedt.

D4Redis

Hvad er det?

Redis er en meget hurtig in-memory key-value store, som ofte bruges som cache.

Hvordan bruger vi det?

SearchApi gemmer søgeresultater i Redis, så gentagne søgninger kan returneres direkte. Begge API-replikaer deler den samme Redis-instans, så cachen virker på tværs af instanserne.

Sig det sådan:
Vi bruger Redis som delt søgecache. Den gør især en forskel ved gentagne søgninger, fordi API'et så kan springe det tunge opslag over og returnere resultatet hurtigere.

D5Cache-aside

Hvad er det?

Cache-aside betyder, at applikationen selv prøver at læse fra cache først, og hvis der ikke ligger noget, henter den data fra databasen og lægger resultatet i cachen bagefter.

Hvordan bruger vi det?

SearchApi bygger en cache-nøgle, prøver Redis først, og ved miss søger den i Postgres og gemmer bagefter resultatet i Redis med en TTL.

Sig det sådan:
Vi bruger cache-aside, fordi det er en enkel og moden strategi. Det betyder, at cache kun bruges, når den hjælper, og databasen stadig er den rigtige backup og sandhedskilde.

D6Cache key

Hvad er det?

En cache key er den nøgle, et resultat gemmes under i cachen. Nøglen skal entydigt repræsentere forespørgslen.

Hvordan bruger vi det?

I SearchApi bygges nøglen ud fra databasevalg, case sensitivity, max amount og de normaliserede søgetermer. Derefter hashes den til en kompakt nøgle som search:....

Sig det sådan:
Cache-nøglen er vores måde at sikre, at to forskellige søgninger ikke får blandet deres resultater sammen.

D7Cache hit og cache miss

Hvad er det?

Et cache hit betyder, at resultatet findes i cachen. Et cache miss betyder, at det ikke gør, og at systemet derfor må hente eller beregne det på anden vis.

Hvordan bruger vi det?

SearchApi logger hit og miss, returnerer cache-status i respons-headeren X-Search-Cache, og registrerer dem som metrics.

Sig det sådan:
Cache hit er gevinsten, fordi vi kan svare hurtigt. Cache miss er stadig okay, for så falder systemet bare tilbage til databasen og fylder cachen bagefter.

D8TTL

Hvad er det?

TTL står for time to live. Det er den tid, et cachet resultat må leve, før det udløber automatisk.

Hvordan bruger vi det?

SearchApi sætter TTL på cachede søgeresultater, så de ikke bliver liggende for evigt. I Compose er TTL konfigureret via SearchCache__TtlSeconds.

Sig det sådan:
TTL er vores simple balance mellem frisk data og god performance. Vi accepterer kortvarig midlertidighed for at få hurtigere svar.
Drift, logs og målinger

Begreber om synlighed i systemet

O1Observability

Hvad er det?

Observability handler om, at man kan forstå, hvad der sker i systemet under drift, uden at skulle gætte.

Hvordan bruger vi det?

I jeres projekt dækker observability både logs, metrics og dashboards. Det giver jer bevis for performance og hjælper med fejlfinding.

Sig det sådan:
Observability gør systemet målbart. Vi kan se både hvor hurtigt det svarer, og hvad der sker, når noget går galt.

O2OpenTelemetry

Hvad er det?

OpenTelemetry er et standardiseret bibliotek til at indsamle telemetri som metrics og tracing fra applikationer.

Hvordan bruger vi det?

I SearchApi er OpenTelemetry koblet på ASP.NET Core, HTTP-kald, runtime-målinger og jeres egne søgemetrics.

Sig det sådan:
OpenTelemetry er den standardmekanisme, der gør det muligt at eksponere målinger fra API'et på en moden måde i stedet for at opfinde vores egen løsning.

O3Prometheus

Hvad er det?

Prometheus er et metrics-system, som scraper målinger fra services over HTTP.

Hvordan bruger vi det?

SearchApi eksponerer et metrics-endpoint, og Prometheus henter målinger som request-adfærd og cache-status derfra.

Sig det sådan:
Prometheus samler tallene ind. Det er herfra vi kan begynde at tale om latency, cache-adfærd og belastning ud fra data i stedet for mavefornemmelse.

O4Grafana

Hvad er det?

Grafana er dashboard-laget. Den viser data fra Prometheus og Loki på en måde, man faktisk kan aflæse under drift.

Hvordan bruger vi det?

I jeres setup kan Grafana bruges til at vise latency, cache-mønstre og logs samlet.

Sig det sådan:
Grafana er ikke der, hvor data bliver skabt. Det er der, hvor drift bliver synlig og nem at forklare.

O5Loki

Hvad er det?

Loki er et loglager, som gør det muligt at samle og søge i logs fra flere containere og services.

Hvordan bruger vi det?

Jeres services kan sende logs til Loki, og Grafana kan derefter vise dem. Det er især nyttigt, når man vil se konkrete fejl, cache-events eller request-forløb.

Sig det sådan:
Loki hjælper os med at se, hvad der faktisk skete. Metrics siger, at noget er langsomt; logs siger ofte hvorfor.
Performance og evaluering

Begreber om hastighed og måling

P1Latency

Hvad er det?

Latency er svartiden for en request, altså hvor lang tid brugeren venter på et svar.

Hvordan bruger vi det?

I projektet er det en vigtig måling, fordi Redis netop skal reducere svartiden ved gentagne søgninger.

Sig det sådan:
Latency er den mest direkte brugeroplevelsesmåling. Den fortæller, hvor hurtigt systemet føles.

P2Throughput

Hvad er det?

Throughput er hvor mange requests systemet kan håndtere i en given periode.

Hvordan bruger vi det?

Det er relevant, når I sammenligner cold cache og hot cache, eller når I forklarer værdien af flere API-replikaer.

Sig det sådan:
Throughput fortæller ikke hvor hurtigt én bruger får svar, men hvor meget samlet arbejde systemet kan klare.

P3P95 og P99

Hvad er det?

P95 og P99 er percentiler. P95 er den svartid, som 95 procent af requests er hurtigere end, og P99 gør det samme for 99 procent.

Hvordan bruger vi det?

De er gode til eksamen, fordi de viser mere end bare gennemsnittet. De fortæller noget om de langsomme requests og den reelle brugeroplevelse.

Sig det sådan:
Vi bruger percentiler, fordi gennemsnit kan skjule problemer. P95 og P99 viser, hvordan systemet opfører sig, når det bliver presset.

P4Cold cache og hot cache

Hvad er det?

Cold cache betyder, at der endnu ikke ligger noget relevant i cachen. Hot cache betyder, at tidligere søgninger allerede har fyldt cachen med nyttige resultater.

Hvordan bruger vi det?

Når I måler performance, er forskellen mellem cold og hot cache central, fordi den viser den konkrete effekt af Redis.

Sig det sådan:
Cold cache viser basisydelsen. Hot cache viser gevinsten, når systemet kan genbruge tidligere arbejde.
Korte eksamenssvar

Gode formuleringer at have klar

Hvorfor Redis?
Redis giver os en delt cache for gentagne søgninger, så begge API-instanser kan svare hurtigere uden at beregne det samme igen.
Hvorfor to API-instanser?
Det viser x-skalering og giver failover, så systemet stadig kan svare, hvis én instans går ned.
Hvorfor Indexer som separat komponent?
Fordi dataopbygning og søgning er to forskellige ansvar, og det gør systemet mere rent og driftsvenligt.
Hvorfor Prometheus og Grafana?
Fordi vi vil kunne bevise performance og ikke kun påstå den.
Er Kubernetes implementeret?
Nej, men arkitekturen er designet, så den kan flyttes derhen senere. Lige nu er Docker Compose det implementerede setup.
Hvad er jeres vigtigste arkitekturvalg?
Stateless API, x-skalering bag Nginx, Redis som cache, separat indexer og observability med metrics og logs.
