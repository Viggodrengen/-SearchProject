SearchProject – eksamens oplæsningsbog

Denne oplæsningsbog er skrevet som et fremlæggelsesmanuskript. Den kan læses højt modul for modul og bruges som grundlag for en podcast eller mundtlig træning.

Rød tråd: Vi viser et søgesystem, der er bygget som et walking skeleton for en microservice-inspireret arkitektur. Fokus er ikke kun kode, men hvorfor ansvaret er fordelt som det er.

Åbning

SearchProject er en søgeplatform, hvor brugeren kan søge i dokumenter via en webapp eller console-klient. Bagved ligger en SearchApi, en Indexer, PostgreSQL, Redis, Nginx og en observability stack med Loki, Grafana og Prometheus.

Det centrale arkitekturvalg er, at SearchApi er stateless og kan køre i flere instanser. Routing ligger ikke i C#-kode, men i infrastrukturen: Nginx i Docker Compose og Ingress/Service i Kubernetes-retningen.

Modul 1–2: Arkitekturprincipper og caseforståelse

Her kobler vi projektet til grundideen i arkitekturprincipper: arkitektur handler om vigtige valg, trade-offs og begrundelser. Vi har ikke bare lavet et program, men et system med tydelige ansvarsområder.

SearchWebApp er UI. SearchApi er søge-use-casen. Indexer bygger indeks. PostgreSQL gemmer state. Redis er cache. Nginx er infrastruktur-routing. Observability er drift.

Modul 3: Skalering – x, y og z

Y-aksen ses i funktionel opdeling: WebApp, API, Indexer, database, cache og observability. X-aksen ses ved at SearchApi kan klones i flere replikaer. Z-aksen er database-retningen: sharding, partitionering eller read replicas.

Vores vigtigste konkrete skalering er x-akse på SearchApi. I Docker Compose kører searchapi1 og searchapi2 bag Nginx. I Kubernetes bliver det pods bag en Service.

Modul 6: Z-scale og databaseopdeling

Projektet implementerer ikke fuld sharding, men vi kan forklare retningen. Hvis databasen bliver flaskehals, kan vi dele data efter tenant, datadomæne eller dokumenttype. Alternativt kan vi bruge read replicas for læsetunge søgninger.

Eksamen: Sig tydeligt at z-akse er en arkitekturplan, ikke en implementeret feature.

Modul 7–8: Operations, miljøer og deployment

Docker Compose er vores stabile demo-miljø. Det starter Postgres, Redis, SearchApi-replikaer, Nginx, WebApp og observability. Kubernetes er produktionsmodellen.

I Kubernetes bruger vi Ingress, Service, Deployment, Pods, HPA, StatefulSet/PVC og Secrets. Control Plane tegnes ikke, fordi det er platformens styringslag.

I cloud kan Kubernetes kobles til cloud-providerens load balancer. Lokalt/on-prem kan man bruge Nginx Ingress Controller, Traefik, MetalLB eller minikube tunnel.

Modul 9: Performance og metrics

Vi bruger performance/cache-scriptet til at måle forskel på cold-cache og hot-cache. Pointen er at kunne dokumentere, at Redis faktisk ændrer systemets runtime-adfærd.

Prometheus og Grafana understøtter metrics-retningen, mens scripts giver konkrete tal til eksamen.

Modul 10: Caching

SearchApi bruger cache-aside med Redis. Først spørges cache. Hvis der er hit, returneres resultatet hurtigt. Hvis der er miss, læses data fra PostgreSQL, SearchLogic bygger resultatet, og resultatet gemmes i Redis med TTL.

Trade-off: Cache reducerer latency, men kræver TTL/invalidation-overvejelse.

Modul 11–12: Databaser og Kubernetes state

PostgreSQL er stateful. I Docker Compose bruger vi volumes. I Kubernetes skal state håndteres med PVC, backup og eventuelt StatefulSet eller managed database. Redis kan køre som cache-workload, men cache er mindre kritisk end persistent data.

Diagramgennemgang

Det vigtigste diagram er C4 Container. Det viser systemets runtime-containere og ansvar. Brug det til at forklare y-akse, x-akse, caching og observability.

Det andet vigtige diagram er Kubernetes produktionsdiagrammet. Brug det til at forklare Ingress, Service, Pods, HPA, PVC, Secrets og observability.

Afslutning

Konklusionen er, at projektet er et walking skeleton, der viser de rigtige arkitekturprincipper: ansvar er delt op, SearchApi kan x-skaleres, caching reducerer latency, routing ligger i infrastrukturen, og Kubernetes-diagrammet viser produktionsretningen.
