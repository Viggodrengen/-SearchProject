Åbning

Den røde tråd

SearchProject er et walking skeleton for en søgeplatform. Det centrale er ikke, at systemet er stort, men at det er arkitektonisk forklarbart. Vi har en WebApp, en Console-klient, en SearchApi, en Indexer, PostgreSQL, Redis, Nginx og observability. Hver del har et ansvar, og hvert valg kan kobles til undervisningens arkitekturprincipper.

Den vigtigste ændring i projektet er, at den hjemmelavede C# load balancer er fjernet. I stedet bruger vi Nginx i Docker Compose og beskriver Kubernetes-produktion med Ingress og Service. Det giver en stærk eksamenspointe: platformopgaver som routing og load balancing skal ligge i infrastrukturen, ikke i domænekoden.

Docker Compose
Stabil demo: WebApp → Nginx → SearchApi A/B → Redis/PostgreSQL.
Kubernetes
Produktionsretning: Ingress → Service → Pods/Deployments → Stateful data.
Undervisning
x/y/z-skalering, caching, performance, operations, deployment og databaser.
Modul 1

Introduktion og arkitekturprincipper

Modul 1 handler om, at arkitektur er de vigtige valg og begrundelser. I SearchProject kan vi pege på konkrete valg i stedet for bare at tale abstrakt.

Kobling til projektet

  • SearchWebApp håndterer UI, men ikke søgelogik.
  • SearchApi håndterer søge-use-casen.
  • Indexer håndterer opbygning af indeks.
  • PostgreSQL ejer persistent state.
  • Redis er cache.
  • Nginx er routing-infrastruktur.
  • Loki/Grafana/Prometheus er drift og observability.
Sig det sådan: “Projektets arkitektur handler om at placere ansvar rigtigt. Vi kan forklare, hvorfor hver komponent findes, og hvilken kvalitetsegenskab den understøtter.”
Eksamenspointe: Arkitekturvalg er trade-offs. Vi vælger mono-repo for overskuelighed, men strukturerer det som deployable services.
Modul 2

Caseforståelse og systemets use case

Casen er dokumentsøgning. Det betyder, at vi har to centrale flows: indeksér dokumenter og besvar søgninger.

Projektets domæneflow

  1. Indexer læser dokumenter fra input-mappe.
  2. Indexer skriver ord og dokumentrelationer til PostgreSQL.
  3. Bruger søger via WebApp eller ConsoleSearch.
  4. Nginx sender request til en SearchApi-replika.
  5. SearchApi bruger Redis og PostgreSQL til at returnere SearchResult.
Sig det sådan: “Komponenterne følger use casen. Vi har ikke tilføjet Redis, Nginx eller observability tilfældigt; de løser konkrete runtime-problemer.”
Modul 3

Skalering: x, y og z

Skaleringskuben er en af de vigtigste undervisningskoblinger.

Y-akse

Funktionel opdeling: WebApp, SearchApi, Indexer, database, cache, proxy og observability.

X-akse

SearchApi er stateless og kører som flere replikaer bag Nginx/Service.

Z-akse

Database-retning: read replicas, partitionering eller sharding efter tenant/domæne/tid.

Når jeg viser C4 Containerdiagrammet, vil jeg sige: “Her ser vi y-aksen i opdelingen mellem komponenterne. X-aksen ses ved SearchApi, som er markeret som skalerbar. Z-aksen ligger ikke som implementeret kode, men som databaseplanen for fremtidig vækst.”
Modul 6

Z-scale og splitting databases

Z-skalering betyder, at man deler data i partitioner og router requests til den partition, der ejer data. SearchProject implementerer ikke sharding, men vi har en konkret plan.

  • Først: én PostgreSQL database.
  • Næste trin: read replica til søgeforespørgsler.
  • Senere: partitionering efter datadomæne, dokumenttype eller tenant.
  • Til sidst: egentlig sharding med routingansvar.
Trade-off: Z-skalering giver bedre skalering og isolation, men øger kompleksitet i routing, migration, konsistens og drift.
Sig det sådan: “Vi har valgt ikke at implementere sharding i walking skeleton’et, fordi det ville være overengineering, men vi kan forklare en realistisk migrationsvej.”
Modul 7

Operations, Docker og Kubernetes-introduktion

Docker Compose er vores demo-miljø. Kubernetes er produktionsretningen. Det er vigtigt at kunne forklare forskellen.

Compose

Starter hele stacken lokalt: Postgres, Redis, Indexer, SearchApi A/B, Nginx, WebApp, Loki, Grafana og Prometheus.

Kubernetes

Bruger Ingress, Services, Deployments, Pods, HPA, Secrets og PVC til produktion.

Sig det sådan: “Compose demonstrerer arkitekturen stabilt. Kubernetes-diagrammet viser, hvordan samme ansvar oversættes til produktionsplatformens begreber.”
Modul 8

Deployment, build vs buy, rollback og barrier conditions

Dette modul kobler direkte til beslutningen om Nginx. Vi bygger ikke en platformkomponent selv, når en standardkomponent er bedre.

Build vs buy i projektet

  • Vi bygger SearchApi, fordi det er vores domæne.
  • Vi bruger Nginx til reverse proxy/load balancing.
  • Vi bruger Redis til cache.
  • Vi bruger PostgreSQL til persistent database.
  • Vi bruger Loki/Grafana/Prometheus til observability.

Barrier conditions

  • Build skal være grøn.
  • xUnit tests skal være grønne.
  • Docker Compose config skal validere.
  • Health endpoint skal svare.
  • Performance/failover scripts skal kunne køres.
Sig det sådan: “Build vs buy handler her om at fokusere vores egen kode på søgedomænet og bruge robuste standardkomponenter til infrastruktur.”
Modul 9

Performance, metrics, OpenTelemetry og Prometheus

Performance skal måles, ikke bare påstås. Projektet har scripts og observability til at dokumentere adfærd.

  • `performance-cache-test.sh` sammenligner cold-cache og hot-cache.
  • Prometheus scraper metrics.
  • Grafana viser dashboards.
  • Headers viser cache-status og API-instans.
“Når vi siger, at Redis forbedrer performance, kan vi vise det med testscriptet: cold-cache giver miss, hot-cache giver hit. Det er en konkret måling af arkitekturvalgets effekt.”
Modul 10

Caching med Redis

SearchApi bruger cache-aside. Det betyder, at applikationen selv styrer opslag og skrivning til cache.

  1. Byg cache key fra query, database, case-sensitivity og maxAmount.
  2. Spørg Redis.
  3. Hit: returnér SearchResult.
  4. Miss: læs PostgreSQL, rank resultater, gem i Redis.
Trade-off: Cache reducerer latency og databasebelastning, men kræver TTL og invalidation-overvejelse. I vores demo er TTL tilstrækkeligt, fordi data er relativt statisk.
Modul 11

Databaser i Kubernetes og volumes

PostgreSQL er stateful. Det gør den anderledes end SearchApi.

Stateless

SearchApi pods kan smides væk og genskabes, fordi state ligger udenfor.

Stateful

PostgreSQL kræver volume, backup, restore og muligvis StatefulSet eller managed database.

Sig det sådan: “Kubernetes gør stateless scaling nemt, men stateful data kræver mere disciplin. Derfor er managed database også en realistisk produktionsmulighed.”
Modul 12

Databaser 2, replicas og splitting databases

Dette modul samler database-retningen: read replicas, StatefulSets og splitting databases.

  • Read replica kan aflaste læsetunge søgninger.
  • Partitionering kan dele data efter domæne.
  • Sharding kræver routingansvar.
  • Managed database kan reducere driftsbyrde.
“Hvis systemet voksede, ville vi først måle flaskehalsen. Hvis det er databasen, ville vi starte med read replicas og derefter overveje partitionering eller sharding. Det er z-akse-retningen.”

Typiske spørgsmål

Spørgsmål og korte svar

Hvorfor Nginx?

Fordi routing/load balancing er infrastruktur. C# skal fokusere på søgedomænet.

Er det microservices?

Det er et mono-repo med deployable services. Arkitekturen er microservice-inspireret.

Hvorfor Redis?

For at reducere latency ved gentagne søgninger og aflaste databasen.

Hvorfor Kubernetes?

For at vise produktionsretningen for scaling, deployment, secrets, routing og state.
