Slide 1 / Arkitekturfortælling

SearchProject

  • Walking skeleton for en søgeplatform med web, API, indexer, database, cache og observability.
  • Y-akse: funktionel opdeling i selvstændige services/containere.
  • X-akse: stateless SearchApi kan køre som flere replikaer.
  • Z-akse: database-retning med read replicas, partitionering eller sharding.

Slide 2 / Vigtige valg

Nginx, Redis og observability

Nginx / Kubernetes

Routing og load balancing ligger i infrastrukturen – ikke i C# forretningskode. Docker Compose bruger Nginx; Kubernetes bruger Ingress + Service.

Redis cache

SearchApi bruger cache-aside. Cold-cache giver miss, hot-cache giver hit og lavere latency ved gentagne søgninger.

PostgreSQL

Persistent state. I Kubernetes kræver det PVC/backup eller managed database.

Loki/Grafana/Prometheus

Logs, dashboards og metrics gør arkitekturen observerbar og testbar.

Slide 3 / Eksamen take-away

Hvad viser projektet?

  • Vi kan forbinde kode, Docker Compose og Kubernetes-diagrammer til samme arkitektur.
  • Vi kan begrunde valg: Nginx/Ingress til routing, Redis til caching, Postgres til state.
  • Vi har testet: build, xUnit, Docker Compose config, performance/cache og failover.
  • Konklusion: simpelt projekt, men med tydelige arkitekturprincipper og produktionsretning.
