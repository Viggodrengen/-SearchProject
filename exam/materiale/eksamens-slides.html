SearchProject – Arkitekturprincipper i praksis
Slide 1

Fra søgeprogram til arkitekturfortælling

SearchProject er et walking skeleton for en søgeplatform. Det viser hvordan kode, Docker Compose, Nginx, Redis, PostgreSQL, observability og Kubernetes-diagrammer hænger sammen.

Use case

Indeksér dokumenter og besvar søgninger hurtigt.

Arkitekturvalg

Ansvar er delt mellem UI, API, Indexer, data, cache, proxy og observability.

Eksamen

Vi viser ikke bare kode – vi begrunder valg med undervisningens principper.

Speaker note: Start med at sige at projektet er simpelt med vilje, men arkitektonisk rigt. Det er et walking skeleton, hvor vi kan pege på konkrete valg.
Slide 2

C4 Container: ansvar og y-akse

Klienter

SearchWebApp og ConsoleSearch kalder samme API endpoint. UI er adskilt fra søgelogik.

Domæneservices

SearchApi søger og ranker. Indexer bygger indekset. De har forskellige ansvar.

Infrastructure

Nginx router. Redis cacher. PostgreSQL gemmer state. Loki/Grafana/Prometheus observerer.

Y-akse = funktionel opdelingMaintainabilityTydelige servicegrænser

Speaker note: Vis C4 Containerdiagrammet. Forklar at Y-aksen er opdelingen i ansvar. Det er ikke bare mapper; det er deployable runtime-komponenter.
Slide 3

X-akse: stateless SearchApi bag Nginx

SearchApi kan køre som flere ens replikaer, fordi persistent state ligger i PostgreSQL og cache ligger i Redis.

Docker Compose

Nginx fordeler trafik til searchapi1 og searchapi2.

Kubernetes

SearchApi ville køre som pods bag en Service, evt. skaleret med HPA.

Failover

Failover-scriptet stopper én API-instans og verificerer fortsat trafik via den anden.

Speaker note: Fremhæv beslutningen: Vi fjernede C# load balancer. Nginx/Ingress/Service er det rigtige lag til routing.
Slide 4

Build vs buy: hvorfor Nginx?

Build

Vi bygger SearchApi, fordi søgelogikken er vores domæne.

Buy/use

Vi bruger Nginx, Redis, PostgreSQL, Grafana, Loki og Prometheus som standardkomponenter.

Argument

Platformopgaver skal ikke opfindes i C# for et eksamensprojekt. Det ville være overengineering.

Build vs buyBarrier conditionsRollback/operations

Speaker note: Kobl til modul 8. Et godt valg er ikke altid at bygge mere kode. Nogle gange er det professionelle valg at fjerne kode.
Slide 5

Redis caching og performance

SearchApi bruger cache-aside. Gentagne søgninger kan returneres fra Redis i stedet for at slå op og ranke igen.

  1. Byg cache key fra query + options.
  2. Spørg Redis.
  3. Hit: returnér SearchResult.
  4. Miss: læs PostgreSQL, rank, gem i Redis med TTL.
Speaker note: Kobl til modul 9 og 10. Performance er målt med scripts: cold-cache giver miss, hot-cache giver hit.
Slide 6

Kubernetes produktionsretning

Ingress

Ekstern HTTP/HTTPS indgang. Kan implementeres af Nginx Ingress Controller.

Service

Stabilt internt endpoint og load balancing til pods.

Deployment + Pods

SearchApi kører som flere pods og kan skaleres med HPA.

State

PostgreSQL kræver PVC/backup eller managed database.

Speaker note: Forklar cloud vs lokal/on-prem: cloud load balancer, Nginx Ingress Controller, Traefik, MetalLB eller minikube tunnel.
Slide 7

Z-akse og databaseplan

Z-skalering er ikke fuldt implementeret. Den er vores realistiske database-retning.

Nu

Én PostgreSQL database med dokumenter og indeks.

Næste trin

Read replicas til læsetunge søgninger.

Senere

Partitionering/sharding efter tenant, domæne, dokumenttype eller tid.

Speaker note: Vær ærlig: Vi har ikke implementeret sharding, fordi det er for meget for walking skeleton. Men vi kan forklare trade-off og migrationsvej.
Slide 8

Konklusion og testbevis

Verificeret

Build, xUnit tests, Compose config, smoke test, performance/cache og failover.

Arkitekturprincipper

x/y/z, caching, observability, deployment, build vs buy og stateful vs stateless.

Take-away

Vi har et simpelt system, men med en klar produktionsretning og tydelige begrundelser.

Speaker note: Slut med at gentage: ansvar er placeret rigtigt. Forretningslogik i SearchApi, routing i infrastruktur, state i database, cache i Redis, drift i observability.
