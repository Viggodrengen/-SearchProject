Slide 1 / 3 • ca. 3 min

Løsningen og de bevidste valg

Målet var et walking skeleton, der opfylder eksamenskrav om skalering, monitorering, caching og database-retning. Vi brugte rød tråd: problem → valg → bevis.

Problem

Intern søgning skal indeksere dokumenter og levere hurtige svar stabilt.

Valg

SearchWebApp, SearchApi, Indexer, Nginx, Redis, PostgreSQL, observability.

Nøglebeslutning

Custom C# load balancer fjernet → Nginx routing/load balancing (build-vs-buy).

Bevis

Build/test grøn, failover test, performance/cache test, driftsoverblik i dashboards.

Teoriforankring

M1-3: ansvar + X/Y/Z • M7-8: drift/deploy • M9-10: performance/cache • M11-12: DB-state.

X: 2 API-instanserY: funktionel opdelingCache-aside med Redis

Speaker note: Start med at forankre løsningen i eksamenskravene: x-skalering, monitorering, caching og DB-retning. Peg derefter i C4-diagrammet på request-flowet (klient → Nginx → SearchApi → Redis/Postgres) og forklar ansvarsplacering. Fremhæv nøglebeslutningen: custom C# load balancer blev fjernet, fordi routing er platformansvar. Slut denne slide med en trade-off-sætning: “Vi fik renere domænekode, men accepterer mere infrastrukturkonfiguration.”
Slide 2 / 3 • ca. 2.5 min

Ønsket produktion i Kubernetes: hvorfor og hvilke trade-offs?

Vi går fra demo-runtime (Compose) til produktionsruntime (Kubernetes) med samme arkitekturlogik men højere driftsmodenhed.

Hvorfor K8s?

  • Standardiseret deployment
  • Skalerbar drift (Deployments/HPA)
  • Tydelig routing (Ingress/Service)
  • Bedre operabilitet ved vækst

Ulemper

  • Højere kompleksitet i drift
  • Stateful data er sværest
  • Kræver platformkompetencer

Z-retning

  • Nu: single DB
  • Næste: read replicas
  • Senere: partitionering/sharding
  • Trinvis modenhed frem for big bang
Speaker note: Forklar først hvad der overføres 1:1 fra Compose: samme serviceansvar, samme flow, men nyt driftlag. Derefter hvorfor Kubernetes vælges: standardiseret deployment, robust routing, skalerbarhed og operationel styrbarhed. Vær så eksplicit om omkostninger: stateful data, backup/restore og platformkompetence er de svære områder. Slut med z-roadmap: read replicas først, partitionering senere, sharding kun hvis målinger kræver det.
Slide 3 / 3 • ca. 2 min

Kodefokus: hvad der er bygget, og hvorfor det er godt

Struktur

Interfaces / Repository / Services
Bevidst enkelt for testbarhed, læsbarhed og lav kompleksitet.

Kerneflow

SearchService + SearchLogic + repository + Redis cache-aside med TTL.

Kvalitetsbevis

xUnit tests, compose-validering, failover-script, performance/cache-script.

Argumentation

Vi bygger domænelogik selv og bruger standardkomponenter til platformansvar.

Afsluttende pointe

  • Valgene er begrundede af teori (modul 1-12) og ikke ad hoc.
  • Vi kan koble hver kodebeslutning til konkrete undervisningsbegreber.
  • Løsningen er demonstrerbar nu og har realistisk K8s-evolution.
  • Vi kan forklare både styrker, svagheder og næste arkitekturtrin.
Speaker note: Brug de sidste minutter på at vise, at arkitekturen findes i koden: SearchService orkestrerer flow, SearchLogic indeholder domæneadfærden, Repository abstraherer dataadgang. Vis kort cache-aside forløbet (hit/miss) og knyt det til performancebeviset. Afslut med en moden formulering: “Vi har bevidst valgt en enkel kodearkitektur for klarhed, testbarhed og hurtig forklarbarhed i drift.”
